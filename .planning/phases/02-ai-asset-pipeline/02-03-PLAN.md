<objective>
Expose AI pipeline outcomes clearly in the product UI and align docs/settings with real provider behavior.

Purpose: Phase 2 is incomplete if backend orchestration exists but users cannot understand partial outcomes or how to recover manually.

Output: Query + detail page updates for run visibility, create-form copy aligned to provider priority, settings/readme documentation updates, and safe UX messaging for partial/failure states.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| persisted run summary -> rendered UI | Provider-derived text can become user-visible in detail page feedback. |
| query params -> banner copy | Redirect feedback may carry untrusted strings. |
| docs/settings copy -> operator behavior | Inaccurate docs can cause unsafe assumptions about key handling and provider capability. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-02-03-01 | Information Disclosure | `/contents/[id]` feedback rendering | mitigate | Reuse/extend safe display guards so run summaries and query messages never render stack traces, key-like tokens, raw JSON payloads, or local paths. |
| T-02-03-02 | Tampering | query-param-driven banners | mitigate | Treat search params as display-only and normalize through safe-message decoder before rendering. |
| T-02-03-03 | Repudiation | provider status visibility | mitigate | Show latest provider, run status, and task id (if present) in deterministic UI fields tied to persisted run records. |
| T-02-03-04 | Spoofing | documentation about provider behavior | mitigate | Update settings/readme copy to reflect Manus primary + Gemini fallback accurately; remove stale mock-only language. |
</threat_model>
